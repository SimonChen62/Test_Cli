# CalliLens UI 重整任务书

## 1. 本轮目标

本轮把 CalliLens 从临时 Demo 改成可答辩的三段式体验：

1. 首页：只呈现 CalliLens 标题和极短副标题，通过滚动进入功能选择。
2. 网页已存库：展示当前主样例 `work_003`，避免多个样例分散答辩主线。
3. 核心导览：围绕同一幅作品完成第一印象、人工观察点、形式证据、局部探针和反思表达。

本轮不做自动书法评分。系统定位是：算法辅助观察，人工确认解释，保留审美开放性。

## 2. 页面流程

### 2.1 首页

- 默认访问 `/web/`。
- 首屏只显示：
  - `CalliLens`
  - `书法观察式导览`
- 用户向下滚动后，标题缩小并吸顶。
- 标题下方出现两个入口：
  - `上传个性化书法作品`
  - `网页已存库`

上传入口只作为 V2 占位，不做真实文件上传。

### 2.2 网页已存库

- 点击 `网页已存库` 后进入作品库状态。
- 作品库只展示 `work_003`：
  - 赵孟頫《行书光福重建塔记卷》右段题名与正文局部
- 点击 `进入观察` 后进入核心导览界面。
- `work_001` 继续保留在数据文件中，但不在本轮 UI 展示。

### 2.3 核心导览

核心界面分为左右两栏：

- 左侧：作品查看器。
  - 默认显示干净原作。
  - 可切换原作、气脉、墨迹、留白、虚实、笔墨图层。
  - 点击图像任意位置可生成竖向局部探针。
- 右侧：纵向工作流。
  - 第一印象
  - 观察点
  - 形式证据
  - 局部探针
  - 我的反思

观察点由人工标注提供，图像上只显示透明虚线圈或框，避免把标注误解为红线描摹或算法判定。

## 3. 前端状态

前端使用四个页面状态：

- `home`：首页。
- `library`：网页已存库。
- `upload`：上传占位页。
- `demo`：核心导览页。

直达链接仍然保留：

```text
/web/?work=work_003&view=demo
```

## 4. 为什么本轮不做后端

本轮不搭 MySQL，也不做账号系统，原因如下：

- 当前答辩重点是“如何帮助新手观察书法”，不是用户管理。
- 只有一个主样例，静态 JSON 足够支撑演示。
- 现在上数据库会增加部署、同步、权限和数据一致性风险。
- 自动上传后还需要图像预处理、图层生成、人工校正流程，不能只靠数据库解决。

V2 如果要做上传，建议路线是：

```text
FastAPI + SQLite + uploads/
```

SQLite 比 MySQL 更适合课堂 Demo：部署轻、文件可随项目走、后续也能平滑迁移。

## 5. 小组分工建议

- 前端同学：
  - 维护三段式页面流程。
  - 保证移动端不重叠、不溢出。
  - 继续优化观察点、详情、反思区的交互。
- 数据/标注同学：
  - 维护 `data/work_003/annotation.json`。
  - 每个观察点必须写清形式证据、观看感受、审美解释。
  - 气脉标注只圈代表性区域，不需要每两个字都标。
- 图像处理同学：
  - 维护 OpenCV 输出图层。
  - 输出原作、二值图、骨架图、笔画粗细、墨色密度、留白候选。
- 答辩同学：
  - 统一表述：算法只提供可见证据，审美解释由人工导览和用户反思完成。

## 6. 验收标准

- `/web/` 默认只看到极简首页。
- 下滚后标题缩小吸顶，并出现两个功能框。
- 点击 `网页已存库` 后只显示 `work_003`。
- 点击 `进入观察` 后进入核心导览。
- 第一印象可填写并进入后续模块。
- 观察卡可点击，图像上出现透明虚线高亮。
- 六个图层按钮可切换。
- 点击图像任意位置会出现竖向探针框和证据报告。
- 反思可填写、提交，并显示专家视角对照。
- 移动端无明显遮挡、按钮溢出或文字挤压。

## 7. 答辩口径

可以说：

- 我们做的是单作品分层观察 Demo。
- OpenCV 负责把墨迹、骨架、留白等可见证据显出来。
- 人工标注负责解释这些证据如何对应气脉、虚实、笔墨。
- 局部探针只给候选提示，不输出审美结论。
- 用户先写第一印象，再对照导览和专家解释，目的是学习“如何看”。

不要说：

- 系统能自动判断书法好坏。
- 系统能自动准确识别气韵。
- 上传任意作品后已经能完整自动分析。
- 红框或虚线就是唯一正确答案。
